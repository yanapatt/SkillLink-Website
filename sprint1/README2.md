กระบวนการทำงานในการพัฒนา Software ของโครงการ

กระบวนการทำงานใน Sprint ที่ 1

1.  การออกแบบ UI และ Use Case Diagram
       ทีมได้เริ่มต้นด้วยการออกแบบ UI และ Use Case Diagram โดยใช้ Draw.ioเพื่อจำลองการทำงานของแอปพลิเคชัน
    Use Case Diagram ช่วยแสดงความสัมพันธ์ระหว่างผู้ใช้และระบบ พร้อมระบุฟังก์ชันสำคัญที่ผู้ใช้สามารถเข้าถึงได้
      ในการออกแบบ UI มีการปรึกษากันเกี่ยวกับฟังก์ชันที่จำเป็นและการจัดวางที่เหมาะสมสำหรับผู้ใช้
2.  การออกแบบ Model Design
      ทีมได้เริ่มต้นโครงสร้างข้อมูลภายในแอปพลิเคชัน โดยเลือกใช้ Linked Listในการจัดการข้อมูล
      ได้ออกแบบฟังก์ชันพื้นฐาน เช่น
           เพิ่มข้อมูล (Add Task): สำหรับเพิ่มงานใหม่เข้าสู่ระบบ
          ค้นหาข้อมูล (Search Task): สำหรับค้นหาและแสดงงาน
          แสดงข้อมูล (Display Task): สำหรับดูรายการงานทั้งหมด
       เครื่องมือที่ใช้:
        Draw.io: สำหรับออกแบบ Use Case Diagram
     Azure DevOps: สำหรับวางแผนงานและติดตามความคืบหน้า
        Discord: สำหรับการสื่อสารและปรึกษางาน



กระบวนการทำงานใน Sprint ที่ 2

1.  การปรับปรุง Model Design
     ใน Sprint ที่ 2 ทีมได้ขยายฟังก์ชันการทำงานของ 
Linked List เพิ่มเติมตามข้อกำหนดของอาจารย์ ได้แก่:
         ลบงานตามลำดับความสำคัญ (Remove Task by Priority): เพิ่มปุ่มเพื่อลบงานที่มี Priority เฉพาะ
        ลบงานที่เก่าที่สุด (Remove Oldest Task): เพิ่มปุ่มสำหรับลบงานแรกใน Linked List
        ลบงานที่ใหม่ที่สุด (Remove Newest Task): เพิ่มปุ่มสำหรับลบงานสุดท้ายใน Linked List
     การออกแบบฟังก์ชันเหล่านี้ช่วยให้การจัดการข้อมูลเป็นระบบและมีประสิทธิภาพมากขึ้น
2.  การพัฒนาและปรับปรุง UI
     ทีมได้ปรับปรุงการออกแบบ UI โดยใช้ Draw.io และเพิ่ม User Flow เพื่อแสดงขั้นตอนการใช้งานที่ชัดเจน
      มีการทดสอบการทำงานของ UI ควบคู่กับการพัฒนา Model Design
3.  การสื่อสารและประสานงาน**
     จากเดิมที่มีการสื่อสารผ่านแชท ได้เปลี่ยนเป็นการประชุมผ่าน Discord เพื่อความเข้าใจที่ตรงกันมากขึ้น
     มีการปรับ Scrum Meeting เป็น 2 ครั้ง/สัปดาห์
สิ่งที่มีการเปลี่ยนแปลงจาก Sprint ที่ 1

1. ปรับปรุง Model Design:
      เพิ่มฟังก์ชัน Linked List ตามข้อกำหนดของอาจารย์
     ลดความซับซ้อนของโค้ด และปรับปรุงการจัดเก็บข้อมูล
2.  ปรับปรุงการสื่อสาร:
    จากการพูดคุยผ่านแชท เปลี่ยนเป็นการประชุม Discord แบบเรียลไทม์
3.  ปรับ Scrum Meeting:
    จากเดิม 3 ครั้ง/สัปดาห์ลดเหลือ 
2 ครั้ง/สัปดาห์ (อังคาร, พฤหัสบดี)
4.  ตัดระบบที่ไม่จำเป็น:
    ระบบแลกของรางวัล ถูกตัดออกเพื่อเน้นฟังก์ชันหลักให้สมบูรณ์
Requirement ที่เพิ่มเติมใน Sprint ที่ 2

ตามเอกสารอ้างอิงจากอาจารย์ ทีมได้เพิ่มความสามารถของ **Linked List** ดังนี้:
1.  Remove by Priority: ปุ่มลบงานตามลำดับความสำคัญ
2.  Remove Oldest Task: ปุ่มลบงานแรกที่เพิ่มเข้ามา
3.  Remove Newest Task: ปุ่มลบงานล่าสุดที่เพิ่มเข้ามา


สรุปการประชุม 
Retrospective Phase 2
-ปัญหาที่เจอระหว่างการทำงานใน Phase 2
ความเข้าใจเกี่ยวกับ Model Design:
    ทีมยังไม่เข้าใจ Model Design อย่างชัดเจน ทำให้เกิดความล่าช้าในการทำงาน
    แนวทางแก้ไข:ศึกษาทบทวนและลงมือทำอย่างสม่ำเสมอใน Phase ถัดไป
ระยะเวลาไม่เพียงพอ
       เวลาที่มีจำกัดทำให้การทำงานไม่เป็นไปตามแผน
       แนวทางแก้ไข:วางแผนและจัดสรรเวลาให้มีประสิทธิภาพมากขึ้น
ขาดความรู้เกี่ยวกับ Software Design:
     ส่งผลให้เกิดความล่าช้าในการดำเนินงาน
   แนวทางแก้ไข:ทำความเข้าใจเนื้อหาให้ครบถ้วน และปรึกษาร่วมกันก่อนเริ่มงาน
Scrum รายสัปดาห์ไม่เพียงพอ:
     การประชุม Scrum ยังน้อยเกินไป และเวลาว่างของแต่ละคนไม่ตรงกัน
    แนวทางแก้ไข: เพิ่มการ Scrum เป็นมากกว่า 2 ครั้งต่อสัปดาห์

สิ่งที่ดำเนินการได้ดี
การออกแบบ UI และ Use Case Diagram:
    ใช้ Draw.io เพื่อจำลองการทำงานของแอปพลิเคชัน
    มีการปรึกษาและแลกเปลี่ยนความคิดเห็นเกี่ยวกับฟังก์ชันการใช้งาน
   Model Design:
     ออกแบบโครงสร้างข้อมูลภายในแอปพลิเคชันด้วย Linked List
     วางแผนฟังก์ชันที่จำเป็น เช่น การเพิ่ม, ค้นหา, และแสดงข้อมูล

3. สิ่งที่ต้องปรับปรุง 

การสื่อสารล่าช้า:
     เนื่องจากทีมทำงานนอกสถานที่และมีเวลาทำงานไม่ตรงกัน
    แนวทางแก้ไข: เพิ่มการประชุมผ่าน Discord เพื่อประสานงานได้ทันเวลา
    เวลาในการปรับปรุงโครงสร้างข้อมูล:
    ใช้เวลานานเกินไปเนื่องจากขาดความเข้าใจ Model Design
    แนวทางแก้ไข:ศึกษาเพิ่มเติมและวางแผนล่วงหน้าชัดเจน

4. สิ่งที่ได้เรียนรู้ 
การใช้ Draw.io ช่วยให้การออกแบบและการสื่อสารเกี่ยวกับโครงสร้างระบบชัดเจนขึ้น
   การวางแผน Model Designที่เหมาะสมตั้งแต่ต้น ทำให้รู้ขั้นตอนการดำเนินงานชัดเจนขึ้น ทุกคนในทีมรู้หน้าที่ของตัวเอง ทำให้งานสามารถดำเนินได้แม้เวลาว่างไม่ตรงกัน

5. ข้อเสนอแนะ  
 เพิ่มความถี่ในการประชุมผ่าน Discord ให้มากขึ้น เพื่อสื่อสารได้อย่างมีประสิทธิภาพและทันเวลา
วางแผน Scrum Meeting อย่างน้อย 2 ครั้งต่อสัปดาห์ เพื่อติดตามความคืบหน้า

6. สรุป 
การประชุมครั้งนี้แสดงถึงความก้าวหน้าในการออกแบบระบบ โดยมีการปรับปรุงโครงสร้างข้อมูลและตัดฟังก์ชันที่ไม่จำเป็นออก
 เพื่อให้ Phase ถัดไปมีประสิทธิภาพมากขึ้น ทีมควรเพิ่มความถี่ในการประชุม ศึกษา Model Design ให้ชัดเจน และจัดสรรเวลาอย่างเหมาะสม
