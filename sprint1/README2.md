สรุปการประชุม Retrospective Phase 2
-ปัญหาที่เจอระหว่างการทำงานใน Phase 2
ความเข้าใจเกี่ยวกับ Model Design:
    ทีมยังไม่เข้าใจ Model Design อย่างชัดเจน ทำให้เกิดความล่าช้าในการทำงาน
    แนวทางแก้ไข:ศึกษาทบทวนและลงมือทำอย่างสม่ำเสมอใน Phase ถัดไป
ระยะเวลาไม่เพียงพอ
       เวลาที่มีจำกัดทำให้การทำงานไม่เป็นไปตามแผน
       แนวทางแก้ไข:วางแผนและจัดสรรเวลาให้มีประสิทธิภาพมากขึ้น
ขาดความรู้เกี่ยวกับ Software Design:
     ส่งผลให้เกิดความล่าช้าในการดำเนินงาน
   แนวทางแก้ไข:ทำความเข้าใจเนื้อหาให้ครบถ้วน และปรึกษาร่วมกันก่อนเริ่มงาน
Scrum รายสัปดาห์ไม่เพียงพอ:
     การประชุม Scrum ยังน้อยเกินไป และเวลาว่างของแต่ละคนไม่ตรงกัน
    แนวทางแก้ไข: เพิ่มการ Scrum เป็นมากกว่า 2 ครั้งต่อสัปดาห์

สิ่งที่ดำเนินการได้ดี
การออกแบบ UI และ Use Case Diagram:
    ใช้ Draw.io เพื่อจำลองการทำงานของแอปพลิเคชัน
    มีการปรึกษาและแลกเปลี่ยนความคิดเห็นเกี่ยวกับฟังก์ชันการใช้งาน
   Model Design:
     ออกแบบโครงสร้างข้อมูลภายในแอปพลิเคชันด้วย Linked List
     วางแผนฟังก์ชันที่จำเป็น เช่น การเพิ่ม, ค้นหา, และแสดงข้อมูล

3. สิ่งที่ต้องปรับปรุง 

การสื่อสารล่าช้า:
     เนื่องจากทีมทำงานนอกสถานที่และมีเวลาทำงานไม่ตรงกัน
    แนวทางแก้ไข: เพิ่มการประชุมผ่าน Discord เพื่อประสานงานได้ทันเวลา
    เวลาในการปรับปรุงโครงสร้างข้อมูล:
    ใช้เวลานานเกินไปเนื่องจากขาดความเข้าใจ Model Design
    แนวทางแก้ไข:ศึกษาเพิ่มเติมและวางแผนล่วงหน้าชัดเจน

4. สิ่งที่ได้เรียนรู้ 
การใช้ Draw.io ช่วยให้การออกแบบและการสื่อสารเกี่ยวกับโครงสร้างระบบชัดเจนขึ้น
   การวางแผน Model Designที่เหมาะสมตั้งแต่ต้น ทำให้รู้ขั้นตอนการดำเนินงานชัดเจนขึ้น ทุกคนในทีมรู้หน้าที่ของตัวเอง ทำให้งานสามารถดำเนินได้แม้เวลาว่างไม่ตรงกัน

5. ข้อเสนอแนะ  
 เพิ่มความถี่ในการประชุมผ่าน Discord ให้มากขึ้น เพื่อสื่อสารได้อย่างมีประสิทธิภาพและทันเวลา
วางแผน Scrum Meeting อย่างน้อย 2 ครั้งต่อสัปดาห์ เพื่อติดตามความคืบหน้า

6. สรุป 
การประชุมครั้งนี้แสดงถึงความก้าวหน้าในการออกแบบระบบ โดยมีการปรับปรุงโครงสร้างข้อมูลและตัดฟังก์ชันที่ไม่จำเป็นออก
 เพื่อให้ Phase ถัดไปมีประสิทธิภาพมากขึ้น ทีมควรเพิ่มความถี่ในการประชุม ศึกษา Model Design ให้ชัดเจน และจัดสรรเวลาอย่างเหมาะสม
